# Cabin 项目编码规范

## 1. 代码风格规范

### 1.1 基本格式
- **缩进**：使用 2 个空格（不使用 Tab）
- **行长度**：最大 80 字符
- **文件编码**：UTF-8
- **行结束符**：LF (Unix 风格)
- **分号**：使用分号结尾
- **引号**：使用单引号，JSX 属性使用双引号

### 1.2 命名规范
- **变量和函数**：camelCase（如 `desktopItems`）
- **常量**：UPPER_SNAKE_CASE（如 `STATIC_FILE_SYSTEM`）
- **组件**：PascalCase（如 `FileExplorer`）
- **接口和类型**：PascalCase（如 `AppConfig`）
- **文件名**：kebab-case（如 `file-explorer.tsx`）

### 1.3 TypeScript 规范
- **类型优先**：优先使用 interface 而不是 type
- **显式类型**：函数参数和返回值必须显式声明类型
- **泛型约束**：泛型必须有适当的约束
- **可选属性**：使用 `?` 标记可选属性
- **严格模式**：启用 TypeScript strict 模式

## 2. React 开发规范

### 2.1 组件规范
- **函数组件优先**：优先使用函数组件而非类组件
- **Hooks 使用**：必须在组件顶层调用 Hooks
- **Props 解构**：在函数参数中直接解构 props
- **默认 Props**：使用默认参数而非 defaultProps
- **Pure Components**：使用 React.memo 优化性能

### 2.2 Next.js App Router 规范
- **客户端组件标记**：使用 useState/useEffect 的组件必须添加 `"use client"` 指令
- **服务端组件**：无交互的展示组件应保持为服务端组件
- **数据获取**：服务端组件使用 async/await 获取数据
- **路由处理**：使用 App Router 的文件系统路由

### 2.3 状态管理规范
- **局部状态**：使用 useState 管理组件内部状态
- **全局状态**：使用 Context API 管理跨组件状态
- **状态提升**：将共享状态提升到最近的共同父组件
- **避免 prop drilling**：深度嵌套时使用 Context

### 2.4 性能优化规范
- **避免内联函数**：在 render 中避免创建新函数
- **列表 key**：使用稳定且唯一的 key，禁止使用数组索引
- **懒加载**：大型组件使用 React.lazy
- **副作用清理**：useEffect 必须返回清理函数

## 3. 文件系统和目录规范

### 3.1 目录结构
- **功能分组**：按功能而非文件类型组织目录
- **组件就近**：组件相关文件放在同一目录
- **配置集中**：配置文件放在 `app/config/` 目录
- **工具函数**：通用工具函数放在 `lib/` 目录

### 3.2 文件命名
- **组件文件**：`component-name.tsx`
- **工具文件**：`utility-name.ts`
- **配置文件**：`config-name.ts`
- **样式文件**：`component-name.module.css`

### 3.3 导入规范
- **绝对路径**：使用 `app/` 开头的绝对路径导入
- **相对路径**：同目录使用 `./`，父目录使用 `../`
- **第三方库**：直接使用包名导入
- **避免循环依赖**：确保导入关系为 DAG

## 4. 错误处理和调试规范

### 4.1 错误处理
- **防御性编程**：对所有外部输入进行验证
- **错误边界**：关键组件使用 Error Boundary
- **用户友好**：向用户显示友好的错误信息
- **日志记录**：生产环境记录关键错误

### 4.2 调试规范
- **开发日志**：开发阶段可以使用 console.log
- **生产清理**：提交前必须移除所有调试日志
- **类型断言**：避免不必要的类型断言
- **空值检查**：对可能为 null/undefined 的值进行检查

### 4.3 ESLint 规则
- **强制规则**：
  - `no-unused-vars`：禁止未使用变量
  - `no-console`：禁止 console 语句（生产环境）
  - `react/no-array-index-key`：禁止数组索引作为 key
  - `react/react-in-jsx-scope`：JSX 必须引入 React
- **自动修复**：使用 `--fix` 自动修复可修复问题
- **提交前检查**：提交前运行 `npm run lint`

## 5. 测试规范

### 5.1 测试策略
- **单元测试**：核心逻辑和工具函数
- **集成测试**：组件间交互
- **E2E 测试**：关键用户流程
- **快照测试**：UI 组件渲染

### 5.2 测试文件组织
- **同目录测试**：测试文件与被测试文件同目录
- **文件命名**：`component-name.test.tsx`
- **测试覆盖率**：关键模块覆盖率 >= 80%

### 5.3 测试最佳实践
- **测试独立性**：每个测试用例独立运行
- **描述性名称**：测试用例名称清晰描述预期行为
- **避免实现细节**：测试行为而非实现
- **模拟依赖**：使用 mock 隔离外部依赖

## 6. 构建和部署规范

### 6.1 构建流程
- **预构建检查**：构建前运行 lint 和类型检查
- **环境变量**：使用 `.env` 文件管理环境配置
- **静态资源**：图片等资源放在 `public/` 目录
- **代码分割**：利用动态导入进行代码分割

### 6.2 部署规范
- **版本控制**：`.next/` 目录不提交到 Git
- **静态导出**：使用 `next export` 生成静态站点
- **CDN 优化**：静态资源通过 CDN 分发
- **缓存策略**：设置合理的 HTTP 缓存头

### 6.3 性能监控
- **Lighthouse**：定期运行 Lighthouse 性能测试
- **Bundle 分析**：使用 Webpack Bundle Analyzer
- **加载性能**：监控首屏加载时间和交互时间
- **错误监控**：集成错误监控服务

## 7. 文档规范

### 7.1 代码注释
- **JSDoc**：公共 API 使用 JSDoc 注释
- **复杂逻辑**：复杂算法添加行内注释
- **TODO 注释**：使用 `// TODO:` 标记待办事项
- **FIXME 注释**：使用 `// FIXME:` 标记已知问题

### 7.2 README 规范
- **项目概述**：简要描述项目目标和功能
- **快速开始**：提供本地开发的快速启动指南
- **技术栈**：列出主要技术依赖
- **贡献指南**：说明如何贡献代码

### 7.3 API 文档
- **接口定义**：详细描述每个接口的用途和参数
- **示例代码**：提供使用示例
- **版本兼容**：说明版本兼容性要求
- **错误码**：列出可能的错误码和含义

## 8. 安全规范

### 8.1 XSS 防护
- **内容转义**：用户输入必须转义后显示
- **DOM 操作**：避免直接操作 innerHTML
- **URL 验证**：外部链接必须验证和清理
- **Markdown 渲染**：使用安全的 Markdown 渲染器

### 8.2 数据安全
- **敏感信息**：不在客户端存储敏感信息
- **API 调用**：验证所有 API 请求参数
- **CORS 配置**：正确配置 CORS 策略
- **HTTPS**：生产环境强制使用 HTTPS

### 8.3 依赖安全
- **依赖审计**：定期运行 `npm audit`
- **版本锁定**：使用 package-lock.json 锁定版本
- **最小权限**：只安装必要的依赖
- **安全更新**：及时更新有安全漏洞的依赖

## 9. 团队协作规范

### 9.1 Git 工作流
- **分支策略**：使用 feature 分支开发
- **提交信息**：使用约定式提交（Conventional Commits）
- **PR 流程**：代码审查后合并
- **标签管理**：使用语义化版本标签

### 9.2 代码审查
- **审查清单**：
  - 功能正确性
  - 代码可读性
  - 性能影响
  - 安全问题
  - 测试覆盖
- **反馈原则**：建设性、具体、及时
- **自动化检查**：CI 自动运行 lint 和测试

### 9.3 知识共享
- **技术文档**：重要决策记录在文档中
- **代码示例**：复杂模式提供示例代码
- **定期分享**：团队内部技术分享
- **新人引导**：完善的新人入职指南
